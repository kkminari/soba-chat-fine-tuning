개요

배경 및 목적

SOBA 마케팅 리서치 챗봇은 현재 6개 대화 생성 태스크에 Anthropic Claude Sonnet API를 사용하고 있습니다. 외부 API에 대한 비용 부담(월 추정 ~$200)과 서비스 가용성 의존 리스크를 해소하기 위해, 오픈소스 모델 Qwen3-8B를 QLoRA 방식으로 파인튜닝하여 자체 모델로 교체하는 작업을 수행했습니다.

파인튜닝 실험

RunPod A100 80GB 환경에서 3회의 실험을 수행했습니다. 모든 실험은 QLoRA(4-bit NF4 양자화 + LoRA) 방식을 사용하여 전체 파라미터가 아닌 소수의 어댑터 파라미터만 학습했습니다.

QLoRA 방식 개요

QLoRA (Quantized Low-Rank Adaptation)는 대규모 언어 모델을 효율적으로 파인튜닝하는 기법입니다. 원본 모델의 가중치를 4비트로 양자화(압축)하여 GPU 메모리 사용량을 줄이고, 소규모 어댑터 레이어(LoRA)만 학습하여 전체 파라미터의 약 1.8%만 업데이트합니다. 이를 통해 A100 GPU 1장으로 80억 파라미터 모델의 파인튜닝이 가능합니다.

공통 하이퍼파라미터

파라미터값설명
Quantization4-bit NF4모델 가중치를 32비트 → 4비트로 압축. NF4(NormalFloat4)는 정규분포 가중치에 최적화된 양자화 방식
LoRA r32어댑터의 랭크(차원). 높을수록 학습 용량 증가하나 메모리 사용도 증가. 데이터 ~5,000건 규모에 적합한 값
LoRA alpha64어댑터 출력의 스케일링 계수. 일반적으로 r의 2배로 설정하여 학습 신호 강도를 조절
LoRA Dropout0.05 / 0.10학습 시 어댑터 뉴런의 일부를 무작위 비활성화하여 과적합 방지. Exp3a에서 0.10으로 상향
Learning Rate2e-4 (cosine)학습률 0.0002에서 시작하여 코사인 곡선으로 점진 감소. QLoRA 표준 범위 (1e-4 ~ 3e-4)
Warmup Ratio0.05전체 학습의 5%를 워밍업 구간으로 사용. 학습 초기에 학습률을 0에서 서서히 올려 안정적 학습 시작
Batch Size4 x 2 = 8GPU당 4건 처리 x gradient accumulation 2회 = 실효 배치 8. 한번에 8개 샘플의 평균으로 가중치 업데이트
Max Seq Length768 tokens입력+출력의 최대 토큰 길이. 한국어 1글자 ≈ 1.5토큰 기준, 약 500자 대화 처리 가능
Optimizerpaged_adamw_8bit메모리 효율적 옵티마이저. 8비트 양자화된 AdamW로 GPU 메모리 절약
Precisionbf16Brain Float 16비트 연산. A100 GPU에서 지원하는 고속 반정밀도 형식
Target Modulesq/k/v/o/gate/up/down_projLoRA 어댑터를 적용할 레이어. Attention(q,k,v,o)과 FFN(gate,up,down) 전체에 적용하여 학습 표현력 확보

실험 1: Qwen3-14B Baseline

목적: 14B 모델의 기본 성능 측정
결과: Epoch 1에서 최적 성능 도달 (eval_loss 0.6368) 후 과적합 발생. Epoch 1→2→3으로 eval loss가 0.637→0.654→0.711로 상승.
진단: 학습 데이터 4,342건 대비 trainable 파라미터 128.5M이 과다하여 1 epoch만에 패턴 학습 완료. 이후 암기(memorization) 진입.

실험 2: Qwen3-8B Baseline

목적: 14B 대비 8B 모델의 성능 차이 측정 (서빙 비용 절감 가능 여부 판단)
결과: Best eval_loss 0.6777로 14B 대비 6.4% 차이. Token Accuracy는 84.58%로 실질적 차이 미미.
비용 이점: 학습 시간 -31%, 어댑터 크기 -31%, 서빙 GPU 메모리 -43%.

실험 3a: 정밀 체크포인팅 + Dropout 강화

배경 (왜 이 실험이 필요했나):
Exp1, Exp2 모두 "Epoch 1이 best, Epoch 2부터 과적합"이라는 동일한 패턴을 보였습니다. 그런데 체크포인트 저장이 epoch 단위(= Step 543)여서, 실제 최적점이 Epoch 1의 끝인지, 중간 어딘가(예: Step 400? 500?)인지 알 수 없었습니다.

접근:
epoch 단위 저장은 너무 성긴 그물이므로, 100 step 단위로 eval + 저장하여 최적점을 정밀하게 탐색했습니다. 추가로 dropout을 0.05→0.10으로 높여 과적합 진입 속도를 늦추고, epochs를 3→2로 줄여 불필요한 학습 시간을 제거했습니다.

결과:
Step 500(epoch 0.92)이 진짜 최적점으로 확인됨. Epoch 1 끝(step 543)보다 약간 앞이 최적이었으며, Step 600(epoch 1.10)부터는 eval loss가 다시 상승하여 과적합 진입이 확인되었습니다. 학습 시간도 18분으로 Exp1(54분) 대비 66% 단축되었습니다.

향후 계획 (Phase 7: 백엔드 통합)

Phase 6 평가를 통과하여, 최종 모델의 백엔드 통합 작업을 진행합니다. 예상 소요 시간은 15~27시간(작업일 기준 2~4일)입니다.

  • 서빙 환경 구축 4~8시간
    vLLM / RunPod Serverless / HF Endpoints 중 선택. GPU 인스턴스 세팅, 모델 배포, 레이턴시 테스트 (p95 < 3s 목표). merged weights 적용으로 추론 속도 최적화.
  • 백엔드 서비스 구현 3~5시간
    qwen_service.py (어댑터 로드 + 추론 API), llm_service.py (Claude/Qwen 라우팅 인터페이스), 후처리 필터 (title 금지어 제거). use_finetuned_model 플래그로 모델 전환 지원.
  • 통합 테스트 3~4시간
    6개 대화 생성 메서드 (generate_intro, generate_first_question, generate_response, generate_retry_message, generate_ending_message, generate_display_title) 정상 동작 확인. Claude Haiku 분석 메서드 정상 유지 확인. 폴백 테스트 (Qwen 장애 시 Claude Sonnet 전환).
  • 배포 및 모니터링 4~8시간
    스테이징 환경 배포 + QA. 프로덕션 배포 + 24시간 안정성 모니터링. Qwen 추론 레이턴시, JSON 파싱 성공률, 폴백 빈도 모니터링.
