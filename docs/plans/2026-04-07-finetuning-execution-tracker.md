**4.4 수동 샘플링 체크리스트:**

```
태스크별 20건 랜덤 추출 → 각 건에 대해 아래 항목 Pass/Fail 판정

[response 체크리스트]
  □ 코멘트가 사용자 응답에 적절히 공감하는가?
  □ 리프레이징이 원본 질문의 의미를 유지하는가?
  □ 대화체로 자연스러운가? (딱딱하지 않은가?)
  □ 존댓말을 사용하는가?
  □ JSON 구조가 정확한가? (comment, next_question 키)

[intro 체크리스트]
  □ '설문/조사/서베이' 단어가 없는가?
  □ 주제가 자연스럽게 소개되는가?
  □ 소요시간 안내가 있는가?

[first_question 체크리스트]
  □ 첫 질문으로의 전환이 자연스러운가?
  □ 원본 질문의 의미가 유지되는가?

[retry 체크리스트]
  □ 비난 없이 부드럽게 재요청하는가?
  □ 왜 다시 답해야 하는지 설명이 있는가?

[ending 체크리스트]
  □ 감사의 톤이 느껴지는가?
  □ '설문/조사' 단어가 없는가?

[title 체크리스트]
  □ 15자 이내인가?
  □ 원본 의미를 반영하는가?

판정 기준: 불량률 > 10% (12건 이상) → 해당 태스크 프롬프트 수정 후 Phase 3 재실행
```

**4.5 train/val/test 분할 규칙:**

```
분할 기준: seed_id (같은 시드의 모든 데이터는 같은 split에 배치)
  → 학습 데이터에서 본 주제가 테스트에 나오는 데이터 누수(leakage) 방지

비율: train 80% / val 10% / test 10% (seed_id 수 기준)
  → 249 seeds: train ~199 / val ~25 / test ~25

저장 포맷: JSONL (ChatML 형식)
  → processed/train.jsonl
  → processed/val.jsonl  
  → processed/test.jsonl
```

**실험 3 조정 가이드:**

```
문제 진단 → 조정 방법:

loss가 안 내려감
  → lr 올리기 (2e-4 → 3e-4) 또는 r 올리기 (32 → 64)

loss는 내려가는데 eval_loss가 올라감 (과적합)
  → epoch 줄이기 (3 → 2), dropout 올리기 (0.05 → 0.1)

eval_loss는 좋은데 생성 품질이 나쁨
  → lr 줄이기 (2e-4 → 1e-4), epoch 늘리기 (3 → 5)

14B와 7B 차이가 작음 (< 5% 차이)
  → 7B 채택 (서빙 비용 절반)

14B가 확실히 좋음 (> 10% 차이)
  → 14B 채택
```
